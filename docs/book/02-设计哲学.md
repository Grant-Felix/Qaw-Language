## 2.0 核心定位（6 条原则）

Qaw 的设计决策**全部从这 6 条原则推导**。任何新特性、新语法、新工具的取舍，先问"是否符合这 6 条"。

### 原则 1：面向开发者的编程语言

Qaw 把**开发者**放在第一位。语言设计、工具链、文档、社区都围绕"让开发者高效、愉快地工作"展开。

**体现**：

- 错误信息友好，附修复建议（如"是否忘了加 `mut`？"）
- IDE 支持完善（跳转、补全、重构、inline hint）
- 标准库 API 一致、直觉、可猜测
- 文档分层（教程 / API / 原理 / RFC）
- 编译速度快（增量编译 < 1s）
- 社区反馈渠道活跃（GitHub Discussions、RFC 流程）

**反面**：不为"减少语言规则"而牺牲表达力；不为"教学简化"而阉割工业特性。

### 原则 2：代码式编程语言

Qaw 是**代码式**编程语言（code-based），**不**做可视化编程或低代码平台。

**体现**：

- 所有程序都是 `.qaw` 纯文本文件
- 不发明专有的可视化格式或拖拽 IDE
- 鼓励通过代码表达意图（类型系统 + trait 表达设计）
- 与 VCS / code review / refactor 工具链完美兼容
- 支持代码生成（macro / derive / build script）

**反面**：不做"零代码"工具；不与 Excel / 流程图竞争；不做"自然语言编程"。

### 原则 3：高性能、高并发

Qaw 性能目标与 C++ / Rust 同档，并发模型融合 Go (GMP 协程) + Rust (所有权安全)：

| 维度 | 目标 | 验证 |
|:---|:---|:---|
| 编译产物性能 | ≥ Rust 同场景 80%（v0.200） | benchmark suite |
| 启动时间 | < 50ms（CLI 工具） | `qawc --version` |
| 并发协程 | 10k 协程 < 1GB 内存 | stress test |
| 冷编译 | < 5s（中等项目） | `time qawc build` |
| 热编译 | < 1s（单文件改动） | incremental |

**体现**：

- 默认编译到 LLVM → 接近 C/Rust 极限
- 协程 + 通道 + `async/await` → Go 风格高并发
- 编译期所有权检查 → Rust 风格并发安全
- 不为开发便利牺牲运行时性能（无隐藏装箱、无隐式 GC 暂停）

**反面**：不为"简化语法"加运行时开销（如反射、动态派发默认开启）。

### 原则 4：默认禁用不安全与低质量方式

**Qaw 默认所有"不安全"和"低质量"开发方式都处于禁用状态。开发者可以手动打开，但必须明确标注。**

#### 4.1 默认禁用的"不安全"类

| 类别 | 默认 | 打开方式 |
|:---|:---:|:---|
| 裸指针 / 手动内存管理 | ❌ | `@unsafe` 块 |
| 未检查边界（数组越界等） | ❌ | `@allow(bounds_check_off)` |
| 类型擦除（`any` 类型） | ❌ | `@dynamic` 块 |
| 强转绕过类型检查 | ❌ | `@unsafe` 块 |
| 数据竞争 / 共享可变状态无锁 | ❌ | `@unsafe` + `@sync` 块 |
| FFI 跨语言边界 | ❌ | `@extern "C"` 块 |

#### 4.2 默认禁用的"低质量"类

| 类别 | 默认 | 打开方式 |
|:---|:---:|:---|
| `panic!` | ❌ | 必须返回 `Result` |
| `.unwrap()` / `.expect()` | ❌ | `?` 传播 + 错误类型 |
| 全局可变状态（`static mut`） | ❌ | `Mutex` / `RwLock` + 所有权 |
| `goto` / 任意跳转 | ❌ | 显式 `loop` / `break 'label` |
| 隐式类型转换 | ❌ | `as` 显式标注 |
| `eval` 类动态代码执行 | ❌ | 编译期宏 / `comptime` |

#### 4.3 打开机制

每个禁用类别对应：
1. **明确标注**（如 `@unsafe` / `@allow`）
2. **作用域受限**（块级，不允许文件级 / crate 级）
3. **clippy 持续监控**（默认 lint 集合包含禁止项）
4. **教育材料**：每个 `@unsafe` 必须配一段"为什么这里必须 unsafe"的注释

#### 4.4 哲学根源

- **Rust**：所有 `unsafe` 必须显式标注，由社区审计
- **Go**：`go vet` 默认禁止常见错误模式
- **教育**：先学会不犯错，再学会打破规则

### 原则 5：编译为主，解释为辅

**Qaw 是编译型语言（首选部署模式），也是解释型语言（教育/测试工具）。**

#### 5.1 角色分工

| 模式 | 用途 | 后端 |
|:---|:---|:---|
| **编译器**（主） | 生产部署、性能敏感场景 | LLVM IR → 原生机器码 |
| **解释器**（辅） | 教育学习、开发测试、REPL | Tree-walking AST |

两者**共享同一前端**（词法 → 解析 → AST → HIR → 类型检查），仅后端不同。

#### 5.2 工作流

```
开发迭代：
  1. 解释器快速运行（毫秒级）→ 验证逻辑
  2. clippy/lint 静态检查
  3. 编译器构建 → 性能验证
  4. 发布二进制

教学场景：
  1. 解释器 REPL → 立即反馈
  2. 修改源码 → 重新运行
  3. 观察 AST 输出 → 学习语言机制
```

#### 5.3 解释器限制

解释器**不**用于：
- 生产部署（性能不足）
- 嵌入式场景（资源受限）
- 服务器长期运行（GC 不优化）

解释器**专用于**：
- 教学课程
- 单元测试（快速反馈）
- AST 调试（`qawc parse` 输出）

#### 5.4 编译器路线

- v0.1：解释器 POC ✅
- v0.50：编译器可工作（单文件程序）
- v0.100：编译器完整（控制流 + 函数 + 类型）
- v0.150：编译器自举（用 Qaw 写 Qaw）
- v1.0：编译器优化（与 Rust 同档性能）

### 原则 6：多领域通用

Qaw 是**通用编程语言**，不是某个垂直领域的 DSL。覆盖现代软件工程的全部主流场景：

| 领域 | Qaw 的优势 | 关键特性 |
|:---|:---|:---|
| **系统底层与嵌入式** | 零成本抽象 + 严格类型 + 可预测性能 | `no_std` + LLVM 后端 + `#![no_std]` |
| **后端与企业级** | 异步 + 强类型 + 大型代码组织 | `async/await` + trait + workspace |
| **前端与移动应用** | WASM 后端 + JS 互操作 | `qawc build --target=wasm32` |
| **人工智能与数据科学** | 数学库 + GPU FFI + 高性能 | FFI 到 numpy/torch + 数组优化 |
| **游戏开发** | 实时性能 + 图形 API | ECS 模式 + Vulkan/Metal FFI |
| **运维与自动化** | 系统调用 + 单文件可执行 | `std::process` + 静态二进制 |

#### 6.1 设计取舍

- **不为某领域牺牲其他领域**：不做"嵌入式特化"（如无 GC），不做"AI 特化"（如动态类型默认）
- **每个领域有独立包**：避免巨型标准库
- **领域互操作**：通过 FFI / trait 跨领域复用
- **示例覆盖**：每个领域至少 1 个完整 demo

#### 6.2 不可用领域（明确不支持）

| 领域 | 原因 |
|:---|:---|
| 浏览器内嵌脚本（替代 JS） | JS 已是事实标准，无生态优势 |
| 操作系统内核 | 内核需要特殊语言（如 Rust + 汇编），Qaw 不优化此场景 |
| 移动端原生 UI（iOS/Android） | 需要 Apple/Google 平台绑定，与"通用"目标冲突 |

## 2.1 三统原则（设计原则，非核心定位）

除了上述 6 条**核心定位**，Qaw 还有 3 条**设计原则**指导具体决策。

### 书同文：四形等价

每个关键字同时支持四种书写形态：

| 形态 | 示例 | 适用 |
|:---|:---|:---|
| 英文原形 | `func`, `package`, `return` | 国际开发者 |
| 英文缩写 | `fn`, `pkg`, `ret` | 追求简洁 |
| 汉语全拼 | `hanshu`, `baozhuang`, `fanhui` | 中文母语 |
| 拼音首字母 | `hs`, `bz`, `fh` | 极致简洁 |

四种形态在词法层归一化为同一个 Token，源码中可任意混用。

### 车同轨：LLVM + WASM 双后端

同一份源码，跨平台原生产出：

| 后端 | 用途 |
|:---|:---|
| LLVM 17+ | 服务器、桌面、嵌入式 |
| WASM | 浏览器、边缘、插件沙箱 |

### 统一度量衡：三大内存模式

| 模式 | 标注 | 适用 |
|:---|:---|:---|
| 安全模式 | `@safe`（默认） | 90% 场景 |
| 共享模式 | `@shared` | 图、UI 树 |
| 动态模式 | `@dynamic` | 胶水、脚本 |

## 2.2 四大铁律（具体行为准则）

1. **取其精华，去其嫁接**：不发明新轮子。
2. **默认最聪明，特例最自然**：编译器能推导的，绝不让人写。
3. **不拒绝历史，不恐惧未来**：通过 C ABI / Python import 吃掉存量。
4. **极端向后兼容，平滑增量升级**：永不抛弃历史用户。

---

## 2.3 不会做的事（明确非目标）

1. **不挑战 LLVM 的代码生成质量**——我们用 LLVM。
2. **不挑战 V8 的 JS 引擎优化**——我们编译到 WASM。
3. **不做 100% 静态分析保证**——成本过高，cargo/miri 级别即可。
4. **不做"零 unsafe"语言**——系统编程需要底层访问；只是默认禁用 + 显式标注。
5. **不做"AI 自动写代码"**——我们是代码式语言（原则 2），AI 是工具不是替代。

## 2.4 与现有语言的关系

| 语言 | Qaw 取自它的特性 |
|:---|:---|
| C / C++ | FFI、嵌入汇编、`@repr(C)` 布局 |
| Rust | 所有权、生命周期、trait、宏、`@unsafe` 默认禁用哲学（原则 4） |
| Go | `go` 协程、`chan` 通道、`defer`、编译为主+解释为辅的工作流（原则 5） |
| Python | `importPython`、列表推导式、`with` |
| Java / C# | `@attr(...)` 注解 |
| Kotlin / Swift | 字符串插值 `${expr}` 语法 |
| TypeScript | 可选链 `?.`、空值合并 `??` |
