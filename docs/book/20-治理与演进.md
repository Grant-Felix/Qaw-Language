## 20.1 治理架构（Arch Linux 模式）

```
社区（Community）
  ↑ 选举
技术委员会（CTC, Core Technical Committee）
  ↑ 委托
工作组（Working Groups）
  ↑ 参与
贡献者（Contributors）
```

- CTC：5-7 人，任期 2 年，由核心贡献者选举。
- 工作组：编译器、标准库、工具链、运行时、文档、社区。

## 20.2 RFC 流程

1. 提案阶段：在 `docs/rfcs/` 起草（蓝图阶段先在 `docs/decisions/`）。
2. 讨论阶段：至少 15 天社区讨论。
3. 投票阶段：CTC ≥ 2/3 通过。
4. 实施阶段：创建跟踪 issue。
5. 稳定阶段：在 N+1 版本进入稳定。

## 20.3 语义化版本 2.0

`MAJOR.MINOR.PATCH`

| 字段 | 含义 | 兼容性 |
|:---|:---|:---|
| MAJOR | 主版本 | **不**保证兼容 |
| MINOR | 次版本 | 100% 兼容 |
| PATCH | 修订号 | 100% 兼容 |

## 20.4 主版本变更规则

主版本变更**必须**满足：

1. **预告期**：在至少 2 个次版本周期内被 `@deprecated` 警告。
2. **自动迁移**：`yao-upgrade --target-version=N+1` 提供自动化脚本。
3. **兼容模式**：`--compat=<旧主版本号>` 标志**永久保留**。
4. **CTC 全员同意**。

## 20.5 LTS 快照

- 每年 **4 月**发布 LTS。
- 版本号格式：`YYYY.MM-LTS`。
- 维护期：3 年。
- 仅接收安全补丁与 P0 Bug 修复。

## 20.6 自动迁移

```bash
yao-upgrade --target-version=3.0 ./src

# 输出：
# 扫描 128 个文件：
#   - 检测 25 处废弃语法
#   - 检测 5 处 API 变更
#   - 已自动修复 28 处
#   - 2 处需手动修复
```

## 20.7 兼容模式

```bash
yaoc build --compat=1.0 ./legacy_project
```

不修改源码，使用兼容模式编译旧代码。
