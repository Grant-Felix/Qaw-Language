## 21.1 版本方案

格式：**`v{大版本}.{小版本}{dev/beta?}`**

| 字段 | 含义 | 范围 |
|:---|:---|:---|
| 大版本 | 正式大版本（破坏性变更 / 全新架构） | 0, 1, 2, ...，无上限 |
| 小版本 | 变动、更新、修复的**最小不可分割独立改变** | 0, 1, 2, ...，无上限 |
| `dev` / `beta` | 开发或测试中的预发布标签（可选） | 仅在主分支不稳定时使用 |

**跳号规则（严格）**：

- **小版本必须递增 +1**——不允许 0.1 → 0.5 跳号
- **大版本升 +1 时，小版本归零**——这是唯一允许的"看似跳号"
- 任何"为了凑里程碑而跳号"都是错误的，反之亦然（不能用未完成的版本号代表将来要做的事）

**版本号含义 = 已完成的小版本计数**，不是"我们要做 X 个版本的承诺"。

## 21.2 里程碑（达成条件，非时间节点）

每个里程碑是"当以下条件全部成立时达到"。里程碑之间是若干小版本累积，没有预设数量。

| 里程碑 | 达成条件 | 验证方法 |
|:---|:---|:---|
| **v0.50**（参考） | **完整独立的开发语言**：可写不依赖任何外部 C 库的完整程序 | `bootstrap/` 目录下的最小自包含 demo |
| **v0.100**（参考） | **主流编程语言的全部能力**：泛型/trait/异步/FFI/宏等 | 与 Rust/Go 特性矩阵对照 |
| **v0.150**（参考） | **自举**：用 Qaw 写 Qaw 编译器，产物与 Rust 版字节级一致 | 双编译 diff |
| **v0.200**（参考） | **完全替代主流语言**：性能、工具链、DX 不输 Rust/Go/Java | 第三方 benchmark + 用户调研 |
| **v0.260**（参考） | **基础生态**：移植、独立开发、互操作调用其他语言生态 | 至少 N 个生态项目 |
| **v1.0**（确定） | **成熟稳定 + 生态完备 + 远超主流 + 多技术栈单语言项目** | 至少 1 个完整多技术栈项目使用纯 Qaw 实现并生产部署 ≥ 6 个月 |

> 上述数字是参考值。具体达成时的版本号取决于实际完成任务数。  
> 例：如果 v0.50 的任务清单包含 50 个最小独立改变，每完成 1 个 +1，到全部完成时自然落在 v0.50。

## 21.4 工作模式（计划与变化）

**计划（A 模式）**：
- 每季度初，整理当前阶段任务清单
- 标注优先级、依赖关系、预估工时
- 发布到 docs/book/21 + docs/3

**变化（C 模式）**：
- 任务实际完成时，**版本号自然递增**
- 任务新增/删除/重排：直接在文档更新，不需"批准"
- 里程碑达成条件变更：需在路线图变更日志中说明原因
- 重大方向调整：需 RFC 流程（见 docs/rfcs/）

**变更日志**：
本节按时间倒序记录路线图本身的修改。

| 日期 | 变更 | 原因 |
|:---|:---|:---|
| 2026-06-30 | 从时间节点驱动改为任务清单驱动；删除 v0.5 MVP 里程碑；引入 v0.50/v0.100/v0.150/v0.200/v0.260/v1.0 里程碑 | 用户规则："小版本根据项目小变更计数" + "每个大小版本号必须有明确目的和切实可行的计划" |

## 21.5 风险与备选

### 高风险

| 风险 | 影响 | 缓解 |
|:---|:---|:---|
| 任务量超预期 | 里程碑延期 | 路线图变更日志透明记录；接受延期比妥协质量重要 |
| LLVM/Cranelift 集成复杂度 | MVP 阶段延期 | 先评估 Cranelift（更简单），必要时回退到纯 C 后端 |
| 借用检查器正确性证明 | 用户信任受损 | 形式化规范 + 大量 property test + Miri 验证 |
| 自举字节级一致性 | 自举可信度问题 | 严格接受"比特位完全一致"，不妥协 |

### 中风险

| 风险 | 缓解 |
|:---|:---|
| 性能优化困难 | 接受 v0.200 性能 < Rust 80% 的过渡期；持续优化 |
| 生态冷启动 | 提供 FFI + 迁移工具；先做 1-2 个杀手级 demo |
| 社区冷启动 | 主动招募；与高校合作课程；提供商业支持 |
| 文档规模爆炸 | 文档分层（教程/API/原理）；自动生成站 |

### 低风险

| 风险 | 缓解 |
|:---|:---|
| CI 成本 | GitHub Actions + 自托管 runner |
| 商标/域名 | 标准注册流程 |

## 21.6 回退方案

| 场景 | 回退方案 |
|:---|:---|
| LLVM 集成不可行 | 锁定 Cranelift 主版本；或纯 C 后端成熟版本 |
| 自举一致性无法达到 | 保留 Rust 引导版为参考实现；Qaw 版标"自举候选" |
| 标准库功能不够 | 拆分为 std + std-contrib；通过 FFI 依赖成熟库 |
| 性能瓶颈 | 局部模块允许手写汇编；接受"特定场景用 C ABI" |
| 社区停滞 | 调整目标用户群；降低 v1.0 标准 |

## 21.7 路线图治理原则

1. **任务粒度**："最小不可分割独立改变"——每个任务必须有明确可验证的产出
2. **版本自动**：不预先指定"v0.X 做 Y"，让版本号自然反映完成进度
3. **里程碑刚性**：达成条件不打折（v0.50 不允许"差不多完整独立"）
4. **变更透明**：所有路线图本身修改必须记录在变更日志
5. **下游兼容**：v0.X 之间承诺源代码兼容（仅 deprecate，不删除）
